让召回结果变成可回答证据

这份说明把 Q-D Reranker Spec 翻译成中文工程蓝图。核心任务不是判断文档是否“主题相似”，而是学习每个 query-document pair 是否能支撑最终答案，并通过特征、排序模型、回归集和上线门禁形成持续进化的 rerank 系统。

01

目标

构建一个面向 RAG 的 query-document reranker，学习召回文档是否能实际支撑答案，而不是只学习主题或语义相似。

Retriever

第一阶段召回负责高召回率。它的责任是“别漏”，可以接受 topN 中混入一些噪声。

Q-D Reranker

重排器在同一个 query 的候选文档之间竞争排序，把真正能支撑答案的证据提上来，把误导性的 hard negative 压下去。

Generator

生成器只消费重排后的 topK evidence documents，减少无证据回答、错误引用和追问率。

02

核心原则

这些规则是系统边界。后续自动特征发现、模型训练和上线评估都不能绕开。

  1. Candidate snapshots 固定。训练和评估必须比较同一批召回候选，否则无法判断提升来自 reranker 还是 retriever 变化。
  2. 训练、验证、回归、测试硬隔离。回归集是门禁，不是训练材料。
  3. 特征必须在生成前线上可计算。答案引用、LLM post-hoc judge、点击、人工标签只能做 label 或诊断，不能作为模型输入。
  4. 优化 answer evidence，不优化主题相似。同主题但时间错、实体错、版本错、没有证据，都是负例。
  5. 新特征是实验对象。必须经过 ablation、slice evaluation、延迟检查、泄漏检查和 promote/reject 决策。
  6. 每个已修复失败都进入回归集。系统不能反复犯同一个错误。
03

系统流程

系统有两条链路：在线链路负责服务请求，离线链路负责训练、评测、回归和发布。

SVG 流程图 AOnline Path / Offline Path

在线链路

query -> query analyzer -> retriever topN -> candidate snapshot -> feature extractor L0-L2 -> XGBoost LambdaMART reranker -> diversity pass -> topK evidence -> answer generator -> feedback/judgment logs

离线链路

logs + labels + failure cases -> dataset builder -> train/validation/test/regression split -> feature extraction -> ranking model training -> offline evaluation -> regression gate -> shadow evaluation -> A/B rollout

04

V0 非目标

V0 先把可复现、可评测、可回滚的 reranker 做稳，不扩散成全套检索系统重写。

  • 不替换第一阶段 retriever。
  • 不在在线重排热路径使用 LLM scoring。
  • 不直接用 regression cases 训练模型。
  • 不允许没有 feature registry 的散装特征代码增长。
  • 不只优化全局 nDCG，而忽略 query slices 和历史失败。

所有自动生成的特征必须进入 registry/DSL。特征要有版本、输入、表达式、成本层级、线上安全性和预期生效 slice。

06

标签构建

标签不是同等可信的真值。系统要记录来源、置信度、冲突处理和曝光位置。

来源优先级用途
human_label最高golden queries、regression cases、审计样本。
answer_citation_label高引用明确且 chunk 支撑生成 claim 时使用。
llm_judge_label中规模化弱监督，必须用人工标签校准。
user_behavior_label中低点击、展开、复制、停留、点赞点踩，受曝光偏差影响。
retrieval_pseudo_label低主要用于启动阶段和 hard-negative mining。

冲突处理

  • human_label 覆盖所有其他标签。
  • 人工 must_not_include 覆盖引用和用户行为。
  • LLM judge 与引用冲突时，进入 audit 或降低置信度。
  • 用户点击一个已知坏文档，不会把它变成正样本。

标签元数据

{
  "label": 3,
  "label_confidence": 0.95,
  "label_sources": ["human", "answer_citation"],
  "label_rationale": "Chunk directly states Q3 gross margin decline and cost cause.",
  "exposure_position": 4,
  "was_exploration_candidate": false
}
重要边界：label fields 永远不能作为线上模型特征。
07

数据集隔离

训练集教模型学习，回归集防模型忘事。混在一起会背题，指标会漂亮但线上更脆。

SVG 流程图 BSplit Isolation
集合用途
train_set模型拟合与特征学习。
validation_setearly stopping、超参数选择、特征 promotion。
regression_set历史失败门禁，只做 gate。
test_set低频最终冷启动评估。

硬规则

  • 同一个 query_id 不能跨集合。
  • 当 session behavior 参与标签构建时，同一用户会话最好不能跨集合。
  • 同一人工修复失败族不能拆到 train 和 regression。
  • 同一次评估必须复用相同 candidate snapshot。
  • Regression cases 默认不可变，除非明确 retire。

可选 aging policy

Recent data -> validation/shadow emphasis.
Stable historical failures -> regression set.
Old broad logs -> train if labels are still valid.
08

回归集生命周期

回归集应该小、硬、稳定。它不是越大越好，而是要能卡住历史上真正伤人的错误。

创建触发

  • 生产坏答案由 bad rerank 导致。
  • must-include evidence 掉出 topK。
  • must-not-include document 进入 topK。
  • 一次人工排查修复了一个失败模式。
  • 高价值 golden query 发生回归。

生命周期

new_case
  -> active regression gate
  -> periodically reviewed for corpus/version validity
  -> either retained, retired, or generalized
原始回归 case 默认不进训练。如果回归集过大，要把模式泛化成 hard-negative generator，例如从“2024 Q3 vs 2023 Q3”泛化为“时间敏感 query 中，同主题但年份/季度/版本错误的文档作为 hard negative”。

10

Hard Negative 类型

Reranker 真正的价值常常不是找正样本，而是把“看起来很像但不能回答”的文档压下去。

类型定义
semantic_hard_negative向量相似度高，但不能回答 query。
lexical_hard_negative关键词命中高，但上下文错误。
temporal_hard_negative主题匹配，但时间或版本不匹配。
entity_hard_negative同名或相似实体，但实际对象错误。
format_hard_negative导航页、模板、目录、boilerplate。
duplicate_negative重复或近重复 chunk，但证据质量更弱。
stale_version_negative旧政策、旧 spec、旧产品版本，不应回答当前 query。

训练集应包含随机负样本和显式 hard negatives；验证集和回归集必须按 negative type 报告指标。

11

模型训练

推荐第一版用 XGBoost ranking objective。每个 query 是一个 group，候选文档在 group 内竞争。

model:
  type: xgboost_ranker
  objective: rank:ndcg
  eval_metric:
    - ndcg@5
    - ndcg@10
  group_key: query_id
  tree_method: hist
  early_stopping_rounds: 100

训练规则

  • 按 query_id 分组。
  • 不要把 query-doc pair 打散 shuffle 到不同 group。
  • 使用 validation groups 做 early stopping。
  • 模型 artifact 中保存 feature names 和 recipe versions。
  • 模型、特征配置、训练数据 snapshot ID、评估报告必须一起保存。

Baseline 对比

  • baseline_0: dense retriever rank
  • baseline_1: sparse/BM25 rank
  • baseline_2: reciprocal rank fusion
  • baseline_3: existing production reranker, if any
  • candidate: XGBoost Q-D reranker

AutoML / HPO 分工

LLM 负责提出 FeatureRecipe 假设，AutoML/HPO 负责在固定特征集上搜索 XGBoost 参数。不要自己实现 TPE、Bayesian search、ASHA/HyperBand、trial pruning 或分布式 trial 调度。

  • Optuna: V0 默认选择，适合框架自己掌控 snapshot、ablation、gate、report。
  • FLAML: 适合快速、预算敏感的 baseline。
  • Ray Tune: 适合大量 trial、分布式、ASHA/HyperBand。
  • AutoGluon: 适合作为强 tabular AutoML 对照组，不建议作为 XGBoost-only 核心引擎。

HPOEngine Adapter

class HPOEngine:
    def optimize(
        self,
        task_profile,
        feature_set,
        train_snapshot,
        valid_snapshot,
        budget,
    ) -> TrialResult:
        ...

常见搜索维度：max_depth、min_child_weight、eta、subsample、colsample_*、gamma、reg_alpha、reg_lambda、max_bin、early_stopping_rounds。

框架自己实现的治理层

不要重写 AutoML 搜索算法，但要自己实现实验治理和上线门禁。外部 HPO 工具只负责在固定特征集、固定 snapshot 和固定 budget 下给出候选参数。

组件职责
ExperimentRunner管理 trial 生命周期、随机种子、预算、artifact 和可复现性。
FeatureRecipeRegistry校验特征定义、版本、online-safe 标记和 leakage risk。
DatasetSnapshotManager冻结 train、validation、test、regression snapshots。
AblationPlanner构造 A/B/C/D 消融实验，控制 baseline 与 candidate 的 HPO 预算公平。
RegressionGate运行历史失败 case，但不把 regression cases 直接放入训练。
SliceEvaluator报告 query slice、hard negatives 和成本敏感指标。
PromotionGate合并 global、slice、regression、latency、reliability 门禁。
FeatureMemoryStore记录 promoted、rejected、quarantined 特征想法和参数先验。
ReportGenerator生成面向人和 AI agent 的 experiment card。
不要给每个 LLM 新特征都跑完整 HPO。先用 scout 阶段的小预算或固定强参数筛掉弱特征，只对 finalist 特征跑完整 HPO 和 promotion gate。

上线门禁

全局提升不够。候选模型必须通过全局、回归、slice、延迟和可靠性门禁。

SVG 流程图 CPromotion Gate
promotion_gate:
  global:
    ndcg10_min_delta: 0.002
    mrr10_min_delta: 0.000
  regression:
    must_include_recall_at_5_drop_max: 0.000
    must_not_include_at_5_increase_max: 0.000
    golden_mrr10_drop_max: 0.000
  slices:
    max_allowed_ndcg10_drop: 0.010
    critical_slice_max_allowed_drop: 0.000
  latency:
    feature_extraction_p95_ms_max: 20
    model_inference_p95_ms_max: 5
  reliability:
    feature_missing_rate_max: 0.300
    online_safe_required: true
  • 全局提升不够。
  • 回归失败会阻止 promotion。
  • 如果某特征提升 numeric queries 但伤害 policy queries，应改成 conditional feature，而不是全局使用。
14

自动特征发现

自动发现不是让系统自由写特征，而是从失败簇里提出可验证假设，然后进入 FeatureRecipe 和门禁。

SVG 流程图 DFeature Discovery Loop

Discovery Inputs

failed queriespositive documentshard negative documentsquery taxonomyretriever scores/ranksanswer citationsLLM judge labelsuser feedbackhuman annotations

Discovery Loop

  1. 选择一个 failure cluster。
  2. 比较 positive docs 和 hard negatives。
  3. 找出能区分二者的差异。
  4. 生成 DSL 里的 candidate FeatureRecipe。
  5. 离线计算该特征。
  6. 做 ablation：baseline features、baseline + candidate feature、baseline + candidate feature group。
  7. 评估 global、slice、regression、latency。
  8. promote、reject 或 quarantine。
  9. 写入 feature memory。

Feature Discovery + HPO 公平对比

LLM 新特征和 AutoML 调参必须拆开归因。每个候选特征至少比较四组：

实验格含义
Abaseline features + baseline params
Bbaseline features + tuned params
Cbaseline + candidate feature + baseline params
Dbaseline + candidate feature + tuned params

归因方式

  • B - A: 参数搜索收益。
  • C - A: 特征本身收益。
  • D - B: 调参后新特征仍然带来的收益。
  • D - C: 调参对新特征的放大收益。

Promotion 判断

主要看 D - B，不是 D - A。否则系统会把参数搜索带来的提升错误归因给新特征。

LLM 可以做什么

  • 提出“这个 query 需要时间精确匹配”。
  • 指出“负样本文档同主题但季度错误”。
  • 建议尝试 date_granularity_match 或 quarter_overlap。

LLM 不可以做什么

  • 不能用 LLM online score 作为所有候选的 XGBoost 特征。
  • 不能写没有 recipe entry 的散装 Python 特征。
  • 不能使用 answer-generation outputs 作为模型输入特征。

{
  "feature_name": "quarter_overlap",

特征记忆

系统要记住哪些特征有效、哪些无效、哪些暂时隔离。否则自动发现会反复试同一个弱想法。

已经通过门禁的特征。

rejected_features.jsonl

明确失败且记录原因的特征。

quarantined_features.jsonl

有潜力但风险或证据不足的特征。

feature_ablation_runs.jsonl

所有消融实验记录。

在线服务

线上路径只使用生成前可计算的 L0-L2 特征，输出 topK evidence，同时记录可复现日志。

  1. Query analyzer 标注 query type，并抽取 entities、dates、numbers。
  2. Retriever 返回 top100 candidates，包含 dense/sparse scores。
  3. Feature extractor 为每个 query-doc pair 计算 L0-L2 features。
  4. XGBoost ranker 为候选打分。
  5. 可选 diversity pass，移除重复 chunks 和 same-doc crowding。
  6. 返回 top5 或 top10 给 generator。
  7. 记录 ranking、features hash、model version、citations、judge labels、user feedback。

Shadow 与 A/B

新模型先 shadow，不影响真实排序；通过后再有限 A/B，最后全量。

offline pass
  -> regression pass
  -> shadow traffic, no user impact
  -> limited A/B
  -> full rollout

Shadow Metrics

  • rank_delta_distribution
  • must_include_lift_if_labels_available
  • answer_citation_overlap_with_old_model
  • LLM_judge_support_delta
  • latency_delta
  • top_doc_source_distribution_shift

A/B Metrics

  • answer_helpfulness
  • user_followup_rate
  • citation_click_rate
  • thumbs_down_rate
  • unsupported_claim_rate
  • latency_p95

探索机制

如果永远只展示当前模型相信的 topK，反馈会越来越窄。需要少量探索来发现新文档、新 query 类型和新特征。

保守区

Top 5 使用稳定生产排序，保护用户体验。

探索区

Rank 6-10 允许有限探索高不确定性候选。探索候选必须打标，避免后续标签高估生产模型能力。

最小实施计划

先把数据、基线、门禁做起来，再做自动发现。顺序反过来会产出很多不可验证的“聪明特征”。

Milestone 1: Data and Baselines

  • 定义 QueryDocExample、CandidateSnapshot、RegressionCase。
  • 从当前 retriever 导出 candidate snapshots。
  • 构建 dense/sparse/RRF baselines。
  • 创建 100 到 300 条 golden queries 和 50 条 failure regression cases。

Milestone 2: Feature Extraction

  • 实现 V0 feature set。
  • 保存带 feature set version 的 feature matrix。
  • 增加 leakage checks 和 online-safe flags。
  • 增加 feature health report。

Milestone 3: Ranking Model

  • 按 query_id group 训练 XGBoost rank:ndcg。
  • 与 retriever baselines 对比。
  • 增加 HPOEngine adapter，V0 默认使用 Optuna。
  • 先跑 tuned-parameter baseline，再把收益归因给新特征。
  • 输出 global 和 slice metrics。
  • 增加 regression gate。

Milestone 4: Feature Discovery

  • 挖掘 failed query clusters。
  • 生成 FeatureRecipe candidates。
  • 用 scout-stage HPO 低成本过滤弱特征，只对 finalist 特征跑完整 HPO。
  • 比较 A/B/C/D 四组：baseline features/params、tuned params、candidate feature、candidate feature plus tuned params。
  • 运行 ablation 和 promotion gates。
  • 维护 promoted/rejected/quarantined feature memory。

Milestone 5: Serving

  • 增加在线 rerank endpoint/function。
  • 运行 shadow evaluation。
  • 增加 explanation output。
  • 启动有限 A/B。
22

AI Implementation Checklist

给 AI agent 的执行顺序。实现时按这个顺序走，少走弯路。

  1. 先创建 schemas。
  2. 围绕固定 candidate snapshots 构建 dataset builder。
  3. 通过 registry 实现 V0 features，不写散装特征代码。
  4. 用 query_id groups 训练 ranking model。
  5. 增加 HPO adapter，调用现有工具，不自己实现搜索算法。
  6. 先加入 validation 和 regression gates，再做自动特征发现。
  7. gates 存在后才加入 feature discovery。
  8. 对新特征先比较 tuned baseline 和 tuned candidate，再决定 promotion。
  9. A/B 前先做 shadow serving。
  10. 永远不要直接用 regression cases 训练。
  11. 永远不要把 post-generation signals 作为线上模型特征。
  12. 始终按 query slice 和 hard negative type 报告指标。
23

待决问题

实现前需要确认这些边界，否则训练和线上指标可能对不上。

  • 当前 retriever stack 是 dense only、BM25 only，还是 hybrid？
  • Retriever 输出 topN 取 50、100，还是 200？
  • Generator 消费 topK 取 5、8，还是 10？
  • 标签来源优先级是否采用 human > answer citation > LLM judge > user behavior？
  • V0 HPO backend 选 Optuna、FLAML、Ray Tune，还是其他 adapter？
  • scout-stage 和 finalist-stage 的 HPO budget 各是多少？
  • Reranking path 的延迟预算是多少？
  • 当前 document chunks 是否已有稳定 version IDs？
  • Query taxonomy 已存在，还是需要新建？
24

成功标准

V0 成功不是“模型能跑”，而是它能稳定打赢 baseline、不过回归、能解释，并且能让至少一个自动发现特征通过门禁。

  • XGBoost reranker 在 validation nDCG@10 和 MRR@10 上击败 dense/sparse/RRF baselines。
  • Golden query set 不退化。
  • Failure regression set 能阻止已知坏 rerank。
  • Feature extraction p95 latency 在预算内。
  • 至少一个自动发现特征通过 ablation 和 slice gates 被 promote。
  • Debug output 能解释一个文档为什么上升或下降。
