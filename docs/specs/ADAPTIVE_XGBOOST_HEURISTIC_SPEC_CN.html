Adaptive XGBoost · Heuristic Feature Discovery · Regression Gates

一个可迁移的 XGBoost 启发式学习框架

这份 HTML 将 ADAPTIVE_XGBOOST_HEURISTIC_SPEC.md 转成中文工程说明。它把原本的 Q-D Reranker 思路抽象为通用框架，适用于排序、二分类、多分类、回归、计数和部分 time-to-event 场景。核心不是“自动调参”，而是让数据快照、特征配方、模型训练、评测门禁、失败记忆和自动特征发现形成可维护闭环。

01

目标

构建一个自适应 XGBoost 训练框架，让标注样本、历史失败、特征实验和任务反馈持续推动模型进化。

Data snapshot -> feature recipe -> XGBoost model -> evaluation gates -> failure memory -> feature discovery -> controlled promotion

排序

Query-document reranking、搜索排序、推荐排序。目标是 group 内候选竞争，通常用 nDCG、MRR、MAP。

分类

欺诈、流失、风险、意图、工单路由。目标是概率、类别或阈值决策，关注 PR-AUC、logloss、召回和误报。

回归

定价、需求预测、评分、计数和 rate。目标是数值误差、校准、尾部误差和切片偏差。

ranking binary class multi-class regression count/rate survival/time
02

核心原则

这些原则是系统边界。自动发现特征和自动训练都不能绕过这些边界。

  1. 评测快照固定。候选模型必须在同一 data snapshot、feature snapshot 和 task split 上比较。
  2. 训练、验证、回归、测试硬隔离。Regression cases 是门禁，不是训练材料。
  3. 特征必须在预测时可用。未来标签、结果后行为、人工修正和下游决策只能做标签或诊断。
  4. 优化任务效用，不优化泛泛准确率。排序看 evidence quality，分类看错判成本，回归看校准误差或业务损失。
  5. 新特征必须实验化。每个 candidate feature 都要经过 ablation、slice evaluation、成本/延迟、泄漏检查和决策。
  6. 每个已修复失败都进入回归集。历史失败应该成为耐久 gate。
  7. 启发式是系统状态。系统要记住哪些想法有效、哪些失败、在哪些 task profile 下有效。

05

V0 非目标

V0 的重点是可复现、可评估、可回滚的 XGBoost 启发式系统，不是大而全 AutoML。

  • 不尝试成为完整 AutoML 平台。
  • 不替代领域数据收集、标注或产品决策策略。
  • 不直接用 regression cases 训练。
  • 不允许没有 feature registry 的散装特征代码增长。
  • 不只优化单一全局指标，而忽略 slices、历史失败和运行约束。
  • 除非 task profile 明确允许，不在线上热路径使用昂贵 LLM 或人工 review。

四个通用 schema 是框架地基：训练样本、预测上下文快照、回归 case、特征配方。

6.3 RegressionCase

Regression cases 是历史失败门禁，不直接进入训练集。它需要用任务语义表达 expected behavior。

  "entity_id": "transaction_987",

  "expected_behavior": {
    "type": "classification",
    "positive_label": 1,

    "must_not_predict_label": 0
  },
  "failure_type": "false_negative_high_value",
  "slice_tags": ["high_value_transaction", "new_merchant"],
  "notes": "Prior model scored a confirmed fraud transaction below manual review threshold."
}
{
  "case_id": "reg_rank_001",
  "task_profile": "qd_reranking",
  "group_id": "query_001",
  "must_include_entity_ids": ["doc_123::chunk_004"],

Ranking

must_include、must_not_include、rank threshold。

Classification

expected class、score threshold、false-positive/false-negative guard。

Regression

acceptable prediction range 或 max absolute error。

Count / Rate

acceptable bucket、calibration bucket、max relative error。

标签构建

标签来源要分层处理。它们不是同等可信的真值。

来源类型置信度说明
human_label最高golden sets、regression cases、audit samples。
verified_outcome高chargeback、cancel、purchase、default、resolved ticket。
downstream_decision中点击、升级、引用、reviewer 选择类别。
llm_judge_label中可扩展弱监督，必须用人工标签校准。
pseudo_label低用于启动、弱监督和 hard example mining。

冲突处理

  • human_label 覆盖所有其他标签。
  • verified_outcome 覆盖行为代理。
  • 已知 regression expected behavior 覆盖弱标签。
  • LLM judge 和 verified label 冲突时进入 audit 或降低置信度。
  • 用户对错误预测的行为不会让错误预测变正确。

标签元数据

{
  "label": 1,
  "label_type": "binary",
  "label_confidence": 0.98,
  "label_sources": ["chargeback", "human_review"],
  "label_rationale": "Confirmed chargeback and manual fraud review.",
  "exposure_position": null,
  "was_exploration_candidate": false
}
重要边界：label fields 永远不能作为线上模型特征。
08

数据集隔离

训练集教模型学习，回归集防模型忘事。Temporal task 不能随机切分。

SVG 流程图 BSplit Isolation
集合用途
train_set模型拟合与特征选择。
validation_setearly stopping、超参数选择、feature promotion。
regression_set历史失败门禁，只做 gate。
test_set低频最终冷启动评估。

硬规则

  • 同一 entity_id 在有实体历史泄漏风险时不应跨 split。
  • 排序任务同一 group_id 不能跨 split。
  • 使用行为标签或实体历史时，同一 user/session/account 不应跨 split。
  • Temporal task 应按事件时间切分，而不是随机行 shuffle。
  • 同一人工修复失败族不能拆到 train 和 regression。
  • 同一次评估必须复用同一个 context snapshot。
09

回归集生命周期

Regression set 应该小、硬、稳定。它是刹车系统，不是训练语料池。

创建触发

  • 生产事故由模型预测导致。
  • 关键 false positive 或 false negative。
  • 排序 must-include 掉出 topK，或 must-not-include 进入 topK。
  • 回归预测超出可接受范围。
  • 人工调查修复了一个失败模式。
  • 高价值 golden case 发生回归。

生命周期

new_case
  -> active regression gate
  -> periodically reviewed for schema/data validity
  -> either retained, retired, or generalized

原始 regression case 默认不进训练。如果集合过大，把模式泛化成 generator。

Raw case:
  high-value fraudulent transaction was scored low because merchant velocity was missing

Generalized training generator:
  for high-value transactions, create hard positives with sparse merchant history and high velocity indicators

AutoML / HPO 分工

LLM 负责从失败样本、残差、hard examples 和弱切片中提出 FeatureRecipe 假设；AutoML/HPO 负责在固定特征集上搜索 XGBoost 参数。不要自己实现 TPE、Bayesian search、ASHA/HyperBand、trial pruning 或分布式 trial 调度。

  • Optuna: V0 默认选择，适合框架自己掌控 snapshot、ablation、gate、report 和 XGBoost objective。
  • FLAML: 适合快速、预算敏感的 baseline。
  • Ray Tune: 适合大量 trial、分布式执行、ASHA/HyperBand 或集群调度。
  • AutoGluon: 适合作为强 tabular AutoML 对照组；如果系统要保持 XGBoost-only 和 FeatureRecipe-driven，不建议作为核心引擎。

HPOEngine Adapter

class HPOEngine:
    def optimize(
        self,

        feature_set,
        train_snapshot,

常见搜索维度：max_depth、min_child_weight、eta、num_boost_round、subsample、colsample_*、gamma、reg_alpha、reg_lambda、max_bin、scale_pos_weight、early_stopping_rounds。

框架自己实现的治理层

不要重写 AutoML 搜索算法，但要自己实现实验治理和上线门禁。外部 HPO 工具只负责在固定特征集、固定 snapshot 和固定 budget 下给出候选参数。

组件职责
ExperimentRunner管理 trial 生命周期、随机种子、预算、artifact 和可复现性。
FeatureRecipeRegistry校验特征定义、版本、online-safe 标记和 leakage risk。
DatasetSnapshotManager冻结 train、validation、test、regression snapshots。
AblationPlanner构造 A/B/C/D 消融实验，控制 baseline 与 candidate 的 HPO 预算公平。
RegressionGate运行历史失败 case，但不把 regression cases 直接放入训练。
SliceEvaluator报告弱切片、hard examples 和任务成本指标。
PromotionGate合并 global、slice、regression、latency、reliability 门禁。
FeatureMemoryStore记录 promoted、rejected、quarantined 特征想法和参数先验。
ReportGenerator生成面向人和 AI agent 的 experiment card。
不要给每个 LLM 新特征都跑完整 HPO。先用 scout 阶段的小预算或固定强参数筛掉弱特征，只对 finalist 特征跑完整 HPO 和 promotion gate。

自动特征发现

自动特征发现从 failure cluster 或 underperforming slice 开始。LLM 可以提假设，但不能跳过验证。

SVG 流程图 DFeature Discovery Loop

Discovery Inputs

failed casespositive and negative exampleslarge residual casesnear-threshold casesslice-level regressionstask taxonomyupstream scores/ranks/rulesverified outcomesLLM judge labelshuman annotations
  1. 选择 failure cluster 或 underperforming slice。
  2. 比较正确处理样本和失败样本。
  3. 识别可区分差异。
  4. 生成 DSL 中的 candidate FeatureRecipe。
  5. 离线计算特征。
  6. 做 ablation：baseline features、baseline + candidate feature、baseline + candidate feature group。
  7. 评估 global、slice、regression、feature health 和 latency。
  8. promote、reject 或 quarantine。
  9. 写入 feature memory。

Feature Discovery + HPO 公平对比

LLM 新特征和 AutoML 调参必须拆开归因。每个候选特征至少比较四组：

实验格含义
Abaseline features + baseline params
Bbaseline features + tuned params
Cbaseline + candidate feature + baseline params
Dbaseline + candidate feature + tuned params

归因方式

  • B - A: 参数搜索收益。
  • C - A: 特征本身收益。
  • D - B: 调参后新特征仍然带来的收益。
  • D - C: 调参对新特征的放大收益。

Promotion 判断

主要看 D - B，不是 D - A。否则系统会把参数搜索带来的提升错误归因给新特征。

baseline 和 candidate 必须使用相同 data snapshot、feature snapshot、split、metric、regression gate 和 HPO budget。

LLM 可以做

  • 指出 failure cluster 需要 recency feature。
  • 识别高价值 positive 被 sparse history 误伤。
  • 建议 merchant_velocity_24h 或 missing_history_flag。

LLM 不能做

  • 不能未经过预算和门禁就把 LLM online score 作为所有预测特征。
  • 不能写没有 recipe entry 的散装特征代码。
  • 不能把未来 outcome 或 post-decision signal 作为模型输入。

服务链路

预测链路统一由 task profile 决定特征、模型、后处理和决策策略。

  1. Resolve task profile。
  2. Fetch online-safe features 或 batch feature snapshot。
  3. Apply model 使用的 FeatureRecipe versions。
  4. Score with XGBoost model。
  5. Apply optional post-processor：ranking diversity、classification threshold、regression clipping/calibration、decision policy。
  6. Return prediction。
  7. Log model version、feature hash、prediction、decision、later outcome、diagnostics。

Shadow / Backtest / A-B

上线顺序是 offline pass、regression pass、backtest/shadow、有限 A/B 或 staged rollout、全量。

offline pass
  -> regression pass
  -> backtest or shadow traffic
  -> limited A/B or staged batch rollout
  -> full rollout

A/B Metrics By Task

  • Ranking: click/citation/usefulness/support rate。
  • Classification: precision/recall at operating point、manual review load、incident rate。
  • Regression: realized error、calibration、business loss、over/underprediction rate。

探索机制

为避免反馈闭环，在风险允许的任务中保留受控探索。

Ranking

高位保守，低位探索高不确定性候选。

Classification

抽样 near-threshold cases 给人工 review，改善标签。

Regression

审计 high-uncertainty 或 tail predictions，为后续标签质量检查服务。

探索候选必须打标，避免后续标签高估生产模型能力。

最小实施计划

先定义 task profiles 和 schemas，再做特征、训练、门禁和自动发现。

Milestone 1

定义 LearningExample、PredictionContextSnapshot、RegressionCase、FeatureRecipe；至少定义一个 ranking profile 和一个 classification 或 regression profile。

Milestone 2

实现 V0 feature registry，保存 feature matrix version，加 leakage checks、online-safe flags、feature health report。

Milestone 3

按 task-profile objective 训练 XGBoost，支持 ranking groups、classification threshold、regression target transform；增加 HPOEngine adapter，V0 默认用 Optuna，并先跑 tuned baseline 再归因新特征收益。

Milestone 4

挖掘 failed case clusters 和 underperforming slices，生成 FeatureRecipe candidates；用 scout-stage HPO 过滤弱特征，对 finalist 特征跑完整 HPO，并比较 A/B/C/D 消融实验。

Milestone 5

增加 prediction endpoint 或 batch scoring job，运行 shadow/backtest，加入 explanation output 和 staged rollout。

AI Implementation Checklist

给 AI agent 的实现顺序。

  1. 先创建 task profiles。
  2. 模型代码前先创建 schemas。
  3. 围绕固定 snapshots 构建 dataset builder。
  4. 通过 registry 实现 V0 features，不写散装特征代码。
  5. 用 task profile 的 objective 和 metrics 训练模型。
  6. 增加 HPO adapter，调用现有工具，不自己实现搜索算法。
  7. 自动特征发现前先加 validation 和 regression gates。
  8. gates 存在后再加 feature discovery。
  9. 对新特征先比较 tuned baseline 和 tuned candidate，再决定 promotion。
  10. A/B 或全量 batch rollout 前先做 shadow/backtest。
  11. 永远不要直接用 regression cases 训练。
  12. 永远不要把 post-outcome signals 作为线上模型特征。
  13. 始终按 slice 和 hard example type 报告指标。

待决问题

实现前需要明确这些系统边界。

  • V0 task profile 是 ranking、binary classification、multi-class classification，还是 regression？
  • 主指标和 operating point 是什么？
  • critical slices 是哪些？
  • V0 HPO backend 选 Optuna、FLAML、Ray Tune，还是其他 adapter？
  • scout-stage 和 finalist-stage 的 HPO budget 各是多少？
  • 哪些标签可信，哪些是弱标签，哪些只用于诊断？
  • 今天是否已有 data snapshot 和 feature snapshot 系统？
  • 延迟或 batch runtime 预算是多少？
  • entity IDs 和 versions 是否稳定？
  • 任务是否需要 temporal splitting？
  • regression cases 如何创建、review 和 retire？
26

成功标准

V0 成功不是模型能跑，而是它能稳定、可复现、可门禁、可解释地改进一个 task profile。

  • 至少一个 task profile 能从固定 snapshots 训练可复现 XGBoost 模型。
  • Candidate model 在 validation primary metric 上击败 task-specific baseline。
  • Golden/regression cases 不退化。
  • Feature extraction p95 latency 或 batch runtime 在预算内。
  • 指标按 global、slice、hard example type 报告。
  • 至少一个自动发现特征通过 ablation 和 gates 被 promote。
  • Debug output 能解释 prediction、rank 或 score 为什么变化。
